# Workflow - 工作流编排器

**定位**：工作流编排层，负责流程编排和委托，不直接定义编码规范。

用于仓库内开发任务的轻量流程约定。目标是减少误改、过度设计和无验证交付，同时把长期有价值的信息沉淀到项目记忆。

## 架构说明

本 skill 作为**编排层**，负责：
- ✅ 工作流程编排（何时读、何时写、何时验证）
- ✅ 自动检测项目类型
- ✅ 委托给对应的编码规范和框架规范 skills
- ❌ 不直接定义"如何写代码"（由编码规范 skills 定义）

## 自动委托规则

检测到项目类型后，自动加载对应的编码规范：

### 1. 检测语言（编码规范层）
- 发现 `.py` 文件 + `pyproject.toml` / `requirements.txt` → 加载 **`python-oop`**
- 发现 `.ts` / `.tsx` 文件 → 加载 `typescript-standards`（待实现）
- 发现 `.go` 文件 → 加载 `go-standards`（待实现）

### 2. 检测框架（框架规范层，可选叠加）
- Python + 发现 `django` + `ninja` → 加载 **`django-ninja`**
- Python + 发现 `fastapi` → 加载 `fastapi-standards`（待实现）
- TypeScript + 发现 `next.config` → 加载 `nextjs-standards`（待实现）

### 3. 手动指定
用户也可以手动指定 skills 组合：
- `/workflow python` → 只加载 python-oop
- `/workflow django` → 加载 python-oop + django-ninja
- `/workflow` → 自动检测

## 使用时机

当任务满足任一条件时使用本 skill：
- 用户要求处理一个代码仓库、项目目录或已有工程
- 任务涉及多文件修改、重构、修 bug、测试、构建、依赖或部署配置
- 用户要求建立/维护项目记忆、统一工作流、长期规则或团队约定
- 其他专项 skill 要求先遵守仓库级工作流

纯问答、一次性脚本、无仓库上下文的小任务可不强制使用。

## 执行流程

### 1. 智能读取项目上下文

按优先级检测并读取项目配置，找到第一个就停止：

**核心配置（优先级从高到低）：**
1. **`CLAUDE.md`** — Claude Code 官方项目指令文件（优先级最高）
2. **`.cursorrules`** — Cursor 规则文件
3. **`.ai/memory.md`** — 传统项目记忆
4. **`AGENTS.md`** — 多 Agent 协作规范
5. **`.github/CLAUDE.md`** — 团队共享的 Claude 规范

**渐进式加载策略：**

- **小任务**（单文件修改、简单功能）
  - 只读核心配置文件（上述 1-5）
  - 读取直接相关的代码文件

- **中等任务**（多文件重构、新增模块）
  - 核心配置 + 编码约定
  - 检查 `.ai/conventions/` 或 `docs/conventions.md`
  - 查看相关模块的现有实现

- **大型任务**（架构调整、大范围重构）
  - 核心配置 + 编码约定 + 架构文档
  - 检查 `.ai/architecture/`、`.ai/deployment/`
  - 阅读 README、项目文档

**分类记忆支持：**
如果项目使用结构化记忆（推荐），按需读取：
```
.ai/
├── memory.md          # 总览索引（始终读取）
├── architecture/      # 架构决策（大任务读取）
├── conventions/       # 编码约定（中等任务读取）
├── deployment/        # 部署流程（部署任务读取）
└── troubleshooting/   # 故障排查（修 bug 时读取）
```

**注意事项：**
- 不存在配置文件时跳过，不要为了补齐而创建空文件
- 避免一次性读取所有文档，根据任务范围渐进加载
- 如果多个配置文件存在冲突，优先采用优先级高的

### 2. 自动检测并加载编码规范

- 扫描项目文件，识别语言和框架
- 根据”自动委托规则”加载对应的编码规范 skills
- 编码规范定义”如何组织代码”，本 skill 不重复定义

### 3. 明确成功标准

- 用 1-3 句话确认要解决什么、完成后如何判断有效
- 需求明确且低风险时直接推进；不要为了形式化流程频繁打断用户
- 存在关键歧义、明显路线取舍、生产/权限/删除数据等风险时先暂停确认

### 4. 先理解再修改

- 修改前查找已有模式、复用点、命名风格和测试方式
- 复用仓库既有实现优先；新增抽象必须有当前需求支撑
- 保持精确修改：只改完成任务所必需的代码，不顺手重构无关区域
- **编码规范由加载的 skills 定义**（如 python-oop、django-ninja）

### 5. 简单优先实现

- 选择满足需求的最小清晰方案，避免推测性扩展和”以后可能用到”的配置
- 业务逻辑、配置、接口、测试分别放在项目既有边界内
- 清理因本次修改产生的无用 import、变量、文件和说明；不要删除原本无关的历史死代码
- **具体编码细节遵循加载的编码规范 skills**

### 6. 验证并说明

- 优先执行当前环境下最小且相关的验证：目标测试、静态检查、构建、最小运行命令或人工检查
- 测试相关任务应优先跑对应测试；大范围改动再逐步扩大验证范围
- 如果依赖、网络、Docker、权限或时间限制导致无法验证，明确说明未执行项、原因、风险和用户可复现命令

### 7. 智能更新项目记忆

仅在产生长期价值时更新项目配置，根据项目现有结构选择合适的位置：

**更新目标选择（按优先级）：**
1. **`CLAUDE.md`** — 如果项目使用 Claude Code（推荐）
2. **`.cursorrules`** — 如果项目使用 Cursor
3. **`.ai/memory.md`** — 传统项目记忆
4. **分类记忆** — 如果项目使用结构化目录：
   - `.ai/architecture/` — 架构决策
   - `.ai/conventions/` — 编码约定
   - `.ai/deployment/` — 部署流程

**值得记录的信息：**
- ✅ 项目背景、架构决策、目录约定
- ✅ 启动方式、验证命令、部署流程
- ✅ 用户明确偏好、团队规范
- ✅ 重要依赖、服务边界、API 约定
- ✅ 本次任务沉淀出的稳定约定

**不要记录的信息：**
- ❌ 密钥、令牌、密码、私钥
- ❌ 临时日志、构建输出、一次性草稿
- ❌ 具体代码实现（应该在代码中体现）
- ❌ 临时 TODO、最近改动日志

**更新原则：**
- 简单任务没有新增稳定信息时，可以不更新记忆
- 如果项目已有多种配置文件，只更新优先级最高的那个
- 新项目建议创建 `CLAUDE.md`，而不是 `.ai/memory.md`

## 行为准则

- 默认用简体中文沟通；代码标识符和 API 名称遵循项目既有语言
- 不确定时暴露假设和权衡；不要把猜测包装成事实
- 每一行修改都应能追溯到用户请求或必要修复
- 复杂任务先拆成可验证节点；简单任务直接完成
- 高风险操作采取保守策略，涉及生产、权限提升、删除或覆盖数据时显式提示

## 推荐项目配置结构

### 选项 A：单文件配置（适合小项目）

```
项目根目录/
├── CLAUDE.md          # Claude Code 官方配置（推荐）
└── README.md
```

### 选项 B：分类配置（适合中大型项目）

```
项目根目录/
├── CLAUDE.md          # 核心索引和总览
├── .ai/
│   ├── architecture/      # 架构决策记录 (ADR)
│   │   ├── 001-database-choice.md
│   │   └── 002-api-design.md
│   ├── conventions/       # 编码约定
│   │   ├── python.md
│   │   └── api.md
│   ├── deployment/        # 部署文档
│   │   └── production.md
│   └── troubleshooting/   # 故障排查
│       └── common-issues.md
└── README.md
```

### 选项 C：兼容多工具（团队使用不同 AI 助手）

```
项目根目录/
├── CLAUDE.md          # Claude Code 配置
├── .cursorrules       # Cursor 配置
├── .ai/
│   └── memory.md      # 共享记忆（备用）
└── README.md
```

```text
.ai/
├── memory.md
├── context-summary-<task>.md
└── verification-report-<task>.md
```

- `memory.md`：长期稳定项目记忆。
- `context-summary-<task>.md`：复杂任务的上下文摘要，按需创建。
- `verification-report-<task>.md`：复杂验证记录，按需创建。
